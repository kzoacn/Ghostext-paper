\section{Evaluation and Staged Execution Protocol}

This section reports released pilot real-backend outcomes for Q1--Q3 as a template-known operational floor and keeps the same staged protocol for the remaining questions. The objective is still claim-to-evidence traceability: each statement should map to a concrete metric definition, script path, and released artifact.

\subsection{Research Questions and Current Coverage}

The campaign is organized around five questions.

Q1 asks whether deterministic decoding is reliable under realistic prompt and message diversity.
Q2 asks what runtime overhead is introduced by deterministic safety controls.
Q3 asks how much payload efficiency is achieved in packet-carrying tokens and in total transmitted tokens.
Q4 asks how large each approximation term is in practice and which term dominates in different operating regions.
Q5 asks how text-only and traffic-aware detectors behave under single-sample and multi-sample observation.

In this revision, Q1--Q3 have pilot real-backend evidence from 18 end-to-end template-known trials (two runs merged), while Q4--Q5 remain open and are kept as staged execution targets.

The real-backend baseline configuration in these 18 trials is explicitly: \texttt{top\_p=0.995}, \texttt{max\_candidates=64}, \texttt{min\_entropy\_bits=0.0}, and \texttt{total\_frequency=4096}.

\subsection{Metrics with Explicit Definitions}

Let $M$ be message count in a condition, $S$ successful decodes, and $B_i$ packet bytes for sample $i$. Let $T_i^{\mathrm{pkt}}$ be packet-carrying tokens and $T_i^{\mathrm{all}}$ be total transmitted tokens.

Recoverability is
\begin{equation}
\mathrm{SR}=\frac{S}{M}.
\end{equation}

Packet-token efficiency and transmission-token efficiency are
\begin{equation}
\eta_{\mathrm{pkt}}=\frac{\sum_i 8B_i}{\sum_i T_i^{\mathrm{pkt}}},
\qquad
\eta_{\mathrm{all}}=\frac{\sum_i 8B_i}{\sum_i T_i^{\mathrm{all}}}.
\end{equation}

Runtime is reported as median and p90 in the main paper tables, while the released artifacts also retain descriptive p99 values. With $n=18$, the p90/p99 values should be read as order statistics rather than stable tail estimates.

Because retries are sender-side and failed attempts are discarded before transmission, baseline runtime should be interpreted as \emph{total time to final successful cover}, not single-attempt latency.

For approximation analysis, planned reporting remains per-step and per-message summaries of
$\alpha_t$, $N_t/(2F_{\mathrm{tot}})$, and $\beta_t$, together with empirical distribution of
\begin{equation}
\widehat{B}_{\mathrm{TV}}=\sum_t\left(\alpha_t+\frac{N_t}{2F_{\mathrm{tot}}}+\beta_t\right)
\end{equation}
as defined in Equation~\ref{eq:bhat-tv}.

These results support only a narrow pilot claim: all 18 released trials decoded successfully under the tested regime, but the aggregate Wilson 95\% interval $[0.824,1.000]$ should be read literally, so the current evidence is an operational floor rather than a settled reliability estimate. The released artifact still records descriptive p99 encode/decode latencies (215.40 s and 63.92 s), but at $n=18$ these are dominated by a few high-retry cases rather than stable tail behavior and are therefore not headlined in the main paper table. The reported payload-efficiency numbers are therefore best read as pilot measurements under the stated baseline policy, not as general claims about detector resistance, traffic-side indistinguishability, or stability across prompt-regime changes.

For transparency, the aggregate 95\% bootstrap intervals from the released 18-trial sample are $[34.43,57.76]$ seconds for encode median, $[26.17,48.13]$ seconds for decode median, and $[2.33,2.63]$ for mean encode bits/token. Table~\ref{tab:real-baseline-language} now mirrors that reporting style per language cell, but these intervals remain descriptive pilot statistics rather than a substitute for a larger condition-stratified campaign. At $n=9$ per language, the Wilson intervals are mainly reminders of uncertainty; the more informative signal in Table~\ref{tab:real-baseline-language} is the different \texttt{attempts\_used} profile between the two language cells.

Because the released baseline uses \texttt{min\_entropy\_bits=0.0}, the reported 2.478 bits/token should be interpreted as an ungated initial operating point, not as a recommended deployment default. The gap between aggregate mean 2.478 and median 2.586 already indicates non-negligible per-message heterogeneity inside this small pilot, which is another reason not to over-read a single headline figure before the planned $h_{\min}$ sweep.

For language interpretation, the zh condition in this baseline uses CJK prompts and CJK secret messages (see released JSON summaries). The current release does not yet expose full real-backend cover strings in JSONL.

Artifacts are released in this repository under `results/real-backend-baseline-r1/`, `results/real-backend-baseline-r2/`, and merged summaries in `results/real-backend-baseline-summary-merged.{json,md}`. Reproduction commands are provided in `results/real-backend-baseline-reproduce.sh`.

\subsection{Evaluation Axes for Remaining Stages}

For broader coverage beyond the baseline, measurements should follow a factorized grid across message length, prompt family, language, and policy settings. Message length should be stratified by plaintext bytes and packet bytes because cryptographic framing changes effective payload size. Prompt families should include at least narrative, informational, and dialog styles in both English and Chinese.

To address predictable-prompt concerns, each prompt family should include two regimes: private-prompt and template-known. The released baseline covers only the latter regime.

\textbf{Stage 4 (ablation campaign) -- pending:} run one-factor-at-a-time ablations for stability filter, fingerprint check, retry policy, quantization budget, truncation policy, and tail policy.

\textbf{Stage 4.5 (aligned external baselines) -- pending:} run at least one single-condition Ghostext-vs-NLS/OD-Stega comparison on the same model, prompt family, and message set; report decode success, bits/token, and encode/decode latency before any superiority claim.

\subsection{Ablation and Failure Decomposition Protocol}

Ablations should isolate each engineering choice while keeping other knobs fixed. For each ablation, report deltas in recoverability, efficiency, failure composition, and approximation-term summaries.

All failures should be mapped to explicit classes already exposed by the implementation: authentication failure, configuration mismatch, synchronization mismatch, unstable-tokenization exhaustion, low-entropy retry exhaustion, and token-budget exhaustion.

\subsection{Baseline Comparison Block (NLS / OD-Stega)}

Direct Ghostext-vs-NLS/OD-Stega numeric comparison on the same model, prompt family, and message set is not included in the current release and is therefore an explicit gap.

Post-pause, the first comparison table should report at least: decode success, bits/token, encode/decode latency, and detector AUC under $m\in\{1,8,32\}$ aggregation, with aligned token-budget and truncation settings. Until that table exists, Ghostext's positioning relative to NLS/OD-Stega should be read as a systems-integration claim rather than a measured performance comparison.

\subsection{Traffic-Side and Detector Blocks}

Because retries may leak side-channel information through timing and length, each run should log attempt count, wall-clock latency, packet-carrying token count, and total token count. Distributions should be reported both unconditionally and conditioned on retry count.

Detector-facing evaluation should include at least one feature-based classical baseline, one neural text detector, and one LLM-judge-style scorer used only as a perceptual proxy. The same sample sets should be evaluated in single-message and multi-message modes, and the single-message slice is the minimum first release because it is much cheaper than the full accumulation study.

For multi-message mode, performance should be reported as a function of aggregation size $m$ (for example, $m\in\{1,4,8,16,32\}$), not a single pooled number.

\subsection{Statistical Reporting and Artifact Rules}

Each reported metric should include uncertainty information: confidence intervals for rates, bootstrap intervals for medians when appropriate, and full sample counts. The current 18-trial pilot now reports aggregate Wilson/bootstrap intervals together with per-language pilot intervals, but expanded condition grids should report interval estimates per cell. When comparing configurations, report effect sizes with interval estimates rather than significance-only statements.

Negative or null findings should be reported with the same prominence as positive findings.

Each table/figure should have a matching artifact bundle containing raw JSON lines, aggregation scripts, plotting scripts, exact commands, and commit hashes for implementation and paper repositories.

\subsection{Readiness Criteria}

The evaluation campaign should be considered complete only when all five research questions have both summary tables and released raw artifacts, including weak or negative conditions. This criterion prevents selective reporting and keeps claim updates evidence-driven.
